  バグの所在＝upstream

  根本原因は dimos/agents/mcp/mcp_client.py（upstream由来）の設計です：

  - call_tool（:171-186）が画像をツール結果に入れられない制約のため、プレースホルダ文字列だけ返して画
  像は別メッセージで後追い注入する。
  -
  その結果、プレースホルダのターンでモデルが応答してしまい、まだ新画像が無い状態（＝前の画像が残った
  history）で答える。

  これは observe に限らず画像を返すツール全般で起きる upstream の欠陥です。McpClient
  を素で使えば誰でも踏みます。なので「upstreamは悪くなかった」わけではありません。

  ではなぜ fork だけで直せたか

  「直す場所」が upstream ファイルでなくて済んだ、という話です：

  - このブループリントは fork固有の TimedMcpClient（McpClient のサブクラス）を使っている。
  - 修正に必要だったのが return_direct というツール単位のフラグを立てるだけで、それはサブクラスの
  _mcp_tool_to_langchain オーバーライドから設定できた。

  つまり「upstreamのバグを、fork側のサブクラスで上書き回避した」= fork-local
  なワークアラウンドであって、upstream本体の挙動は依然バグったままです。

  整理すると

  ┌───────────────┬──────────────────────────────────────────────────────────────────────────────┐
  │               │                                     評価                                     │
  ├───────────────┼──────────────────────────────────────────────────────────────────────────────┤
  │ バグの原因    │ upstream（mcp_client.py                                                      │
  │               │ の遅延アーティファクト＋プレースホルダで応答する流れ）                       │
  ├───────────────┼──────────────────────────────────────────────────────────────────────────────┤
  │ observe()     │ 正常（最新フレームを返している）                                             │
  │ 自体          │                                                                              │
  ├───────────────┼──────────────────────────────────────────────────────────────────────────────┤
  │ 今回の修正    │ fork側サブクラスでの回避（upstream無編集）。本筋の修正は upstream を直すこと │
  └───────────────┴──────────────────────────────────────────────────────────────────────────────┘

  CLAUDE.md的に upstream は
  fetch専用／PR禁止なので、「fork内で完結する回避策」を選んだ、という位置づけです。もし将来 upstream
  を直接直したい（あるいは本家へ報告したい）なら、call_tool 側で画像を同一ターンに注入する（Command
  等で単一応答化）のが本質的な直し方になります。
