# unitree-go2-agentic-ja E2E Bench Design

Status: implemented 2026-05-16. Live end-to-end verification deferred — Task 6 cannot dry-run without robot (192.168.11.10) and LLM API credentials in this environment. All unit tests pass (12 / 12).

Implementation notes that diverged from this spec:
- Audio injection method changed from "publish to `_audio_subject` directly" to "POST to `/upload_audio` HTTP endpoint" because `JapaneseWebInput` runs in a worker subprocess; the in-process Subject is not reachable through the `ModuleProxy`. The HTTP endpoint already exists on `RobotWebInterface` (port 5555, what the WebUI uses) and runs inside the subprocess that owns the live Subject — so the same code path is exercised. The schema and metrics are unchanged.

See `docs/superpowers/plans/2026-05-16-agentic-ja-bench-e2e.md` for the executed plan.

## 1. Purpose & Scope

### Purpose
Measure end-to-end latency of the `unitree_go2_agentic_ja` blueprint with reproducible fixtures, so that future swaps of STT / LLM / TTS (to cloud managed services or other local providers) can be compared apples-to-apples.

### Scope (MVP)
- Target blueprint: `unitree_go2_agentic_ja` only.
- Event schema is designed to be reusable across blueprints, but other blueprints (e.g. `unitree_go2_agentic_voice_live`) are not wired up in MVP.
- All implementation lives in fork-local files; upstream files are not modified.

### Non-goals (MVP)
- Real microphone / acoustic validation (audio is injected via the existing `audio_subject`).
- Cross-blueprint comparison (deferred — schema-ready but no instrumentation in other blueprints yet).
- Splitting TTS into synth-only vs playback-only timings.
- Cloud provider integration itself (provider swaps happen on other branches; bench is reused).

## 2. Metrics

Two end-to-end latencies are the primary indicators:

| Metric | Definition |
|---|---|
| **e2e-1 (`e2e_response_s`)** | `first_audio_out.t − user_audio_end.t` — time from end of user audio to the first audio chunk of the agent's first `speak` call. |
| **e2e-2 (`e2e_motion_s`)** | `first_motion_tool.t − user_audio_end.t` — time from end of user audio to the first non-`speak` MCP tool invocation. |

`t=0` is defined as the moment the input audio stream ends (the last wav chunk has been published to `audio_subject`). This is reproducible across all systems: VAD tail of any auto-VAD STT provider naturally lands inside `stt_done`, so the metric stays comparable.

"Motion" is defined operationally as `tool != "speak"`. `agentic_ja` exposes no observation-only tools, so this simple split suffices.

Existing breakdown metrics are preserved and made consistent: `stt_done`, `llm_step`, `tools_step`, `turn_done`, plus `mcp_tool:*` (upstream-emitted, correlated by timestamp).

## 3. Event Schema

All bench events are written to `logs/<run-dir>/main.jsonl` via the existing structured logger.

### New events
| `event_kind` | When | Required fields |
|---|---|---|
| `user_audio_end` | replay script, immediately after publishing last audio chunk | `turn_id`, `t`, `audio_seconds`, `fixture_id`, `run_idx`, `warmup` |
| `first_audio_out` | `JapaneseSpeakSkill`, first audio chunk emitted from `_tts_node` after a `speak()` call (per turn, once) | `turn_id`, `t`, `tool="speak"` |
| `first_motion_tool` | `TimedMcpClient`, first non-`speak` `tool_call` observed in an `agent` step output (per turn, once) | `turn_id`, `t`, `tool` |

### Existing events (rewritten to use bench helpers)
- `stt_done` — `web_human_input_ja.py`
- `llm_step` / `tools_step` / `turn_done` — `mcp_client_ja.py`

All fork-local events carry `turn_id` and a monotonic `t` (perf_counter).

### Upstream-emitted events (untouched)
- `MCP tool done` (i.e. `mcp_tool:*`) — no `turn_id`. Analyzer buckets these into turns by timestamp range (between `user_audio_end` and `turn_done` of the same `turn_id`).

### `turn_id` lifecycle
- Generated by `replay_agentic_ja.py` immediately before logging `user_audio_end`.
- Held in a `contextvars.ContextVar` inside `dimos/agents/bench_ja/turn_context.py`.
- Read by all fork-local emitters via `current_turn()`.
- Risk: `contextvars` does not auto-propagate across `Thread` boundaries. Implementation must verify that all relevant emitters share the contextvar with the replay driver, and if not, propagate explicitly. This is a known implementation checkpoint.

## 4. Instrumentation Placement

| Event | File | Approach |
|---|---|---|
| `user_audio_end` | `scripts/replay_agentic_ja.py` (new) | `new_turn()` + `log_bench_event(...)` after last chunk publish |
| `stt_done` | `dimos/agents/web_human_input_ja.py` (rewrite) | Replace ad-hoc `_make_stt_timer` with `turn_context`-aware logging |
| `llm_step` / `tools_step` / `turn_done` | `dimos/agents/mcp/mcp_client_ja.py` (rewrite) | Replace current logger calls with `log_bench_event` |
| `first_motion_tool` | `dimos/agents/mcp/mcp_client_ja.py` | Inspect `tool_calls` on agent-step messages; fire once per turn on first non-`speak` |
| `first_audio_out` | `dimos/agents/skills/speak_skill_ja.py` (rewrite) | `ops.do_action` tap on `_tts_node.emit_audio()`; fire once per `speak()` invocation |
| `mcp_tool:*` | upstream `mcp_server.py` (unchanged) | No `turn_id`; analyzer buckets by timestamp |

`first_motion_tool` fires when the LLM emits the tool_call, not when the MCP server dispatches it. The delta (one MCP RPC roundtrip, ~ms) is negligible for comparison purposes and avoids touching upstream.

## 5. `dimos/agents/bench_ja/` Layout

```
dimos/agents/bench_ja/
├── __init__.py          # re-exports current_turn, new_turn, reset, log_bench_event
└── turn_context.py      # ContextVar + helpers + chokepoint logger
```

`turn_context.py` exposes:
- `new_turn() -> str` — issue an ID (`uuid.uuid4().hex[:12]` — short, collision-free for in-process use; no new dependency), set ContextVar, return.
- `current_turn() -> str | None`
- `reset() -> None`
- `log_bench_event(kind: str, **fields) -> None` — single chokepoint; injects `event_kind`, `turn_id`, `t` (perf_counter), then user fields.

All fork-local emitters route through `log_bench_event` so schema cannot drift.

## 6. Replay Script (`scripts/replay_agentic_ja.py`)

### Responsibilities
1. Build & start `unitree_go2_agentic_ja` in-process.
2. Acquire an `audio_subject` accessor on `JapaneseWebInput` (a new test-facing hook, e.g. `_audio_subject` property).
3. Load fixtures from YAML, iterate (optionally shuffled, with warmup).
4. For each fixture: wait for `agent_idle=True`, publish wav chunks to `audio_subject`, emit `user_audio_end`, wait for `turn_done`.
5. Stop blueprint cleanly.

### CLI
```
python scripts/replay_agentic_ja.py \
  [--fixtures tests/bench_fixtures/agentic_ja/fixtures.yaml] \
  [--runs N]            # repeats per fixture (default 3)
  [--warmup N]          # first N runs flagged warmup=true (default 1)
  [--shuffle]           # randomize order (default off)
  [--realtime]          # publish chunks with sleep (default off, MVP burst-publishes)
  [--out logs/...]      # default logs/{ts}-bench-agentic-ja/
```

### Robustness
- Per-fixture timeout (e.g. 30 s waiting for `turn_done`); on timeout, log an error event and continue.
- `audio_subject` hook is named `_audio_subject` to communicate "internal/test use only".

## 7. Fixture Set

### Layout
```
tests/bench_fixtures/agentic_ja/
├── fixtures.yaml
├── README.md
├── speak_001.wav ... speak_004.wav
├── motion_001.wav ... motion_004.wav
└── both_001.wav ... both_004.wav
```

### Manifest (`fixtures.yaml`)
```yaml
version: 1
fixtures:
  - id: speak_001
    category: speak_only          # speak_only | motion_only | both
    wav: speak_001.wav
    text: "おはよう"
    notes: "短い挨拶"
  ...
```
- `text` is reference metadata; the runtime only consumes the wav.
- 16 kHz mono PCM WAV.

### Initial set: 12 fixtures, 3 categories × 4 each
- `speak_only`: greetings / Q&A / acknowledgements that should yield `speak` only.
- `motion_only`: imperative motion commands.
- `both`: combined commands that should yield both `speak` and motion.

### Generation
- `scripts/gen_fixtures_agentic_ja.py` synthesizes wavs from `text` using `pyopenjtalk` (same engine as the blueprint's TTS — no new deps).
- Idempotent: existing wavs are not regenerated.

### Caveat
TTS-synthesized fixtures may give Whisper an unfair edge over real human speech. Acceptable for MVP; a `tests/bench_fixtures/agentic_ja_human/` set can be added later when cloud STT comparison demands it.

### Commit policy
- Wavs are committed (size on the order of MB; reproducibility outweighs repo bloat).
- `logs/*-bench-agentic-ja/` remains gitignored.

## 8. Analyzer (`scripts/bench_agentic_ja.py`)

Extends the existing analyzer.

### Pipeline
1. Read `main.jsonl`.
2. Bucket events by `turn_id`.
3. Assign `mcp_tool:*` events to turns by timestamp range (event `t` between `user_audio_end.t` and `turn_done.t` of the same turn).
4. Derive per-turn metrics: `e2e_response_s`, `e2e_motion_s`, `stt_s`, `llm_total_s`, `tools_total_s`, `turn_total_s`.
5. Drop turns where `warmup=true`.
6. Aggregate overall and per-category (`speak_only` / `motion_only` / `both`).
7. Per bucket: `n / mean / p50 / p95 / max / min`. Mark buckets with `n < 5` as `[low-n]`.

### Output
- Default: terminal table (e2e first, breakdown second, `mcp_tool:*` third).
- Optional `--json FILE`: machine-readable dump.
- Optional `--md FILE`: markdown report (single run; cross-run compare is future work).

### CLI
```
python scripts/bench_agentic_ja.py [logs/<run-dir>] [--json FILE] [--md FILE]
```

Run directory auto-pick: prefer the latest `logs/*-bench-agentic-ja/`, fall back to `logs/*-unitree-go2-agentic-ja/` for backward compatibility with ad-hoc logs.

## 10. How To Verify

```bash
# initial: generate fixtures (once)
python scripts/gen_fixtures_agentic_ja.py

# run bench
python scripts/replay_agentic_ja.py --runs 3 --warmup 1

# analyze
python scripts/bench_agentic_ja.py
```

Success criteria: terminal output shows `e2e_response_s` and `e2e_motion_s` with non-trivial `n`, broken down per category, plus the existing per-stage breakdown.

## 11. Risks & Open Implementation Questions

1. **ContextVar propagation across threads.** `TimedMcpClient._process_message` may run on a thread distinct from the publisher of `user_audio_end`. Implementation must verify ContextVar propagation works in practice, and explicitly propagate `turn_id` if not.
2. **`audio_subject` hook stability.** Exposing `_audio_subject` couples bench to `JapaneseWebInput`'s internal layout. The leading underscore communicates "internal"; future refactors of `JapaneseWebInput` must re-expose it.
3. **LLM nondeterminism.** A fixture may not always trigger the expected tool. Per-turn metrics will be missing; `low-n` markers surface this in aggregates.
4. **`mcp_tool:*` timestamp bucketing accuracy.** If turns interleave (they shouldn't, since agent is single-threaded per turn), bucketing could misattribute. Add an assertion that no two turns overlap in time.
5. **Fixture-engine self-loop bias.** Whisper transcribing pyopenjtalk-generated audio is likely to be optimistic. Documented as a known caveat; the human-recorded fixture set is the mitigation when needed.
