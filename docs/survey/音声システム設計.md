
## 推奨アーキテクチャ 🎙️

結論は、**“常時ASR”ではなく、状態機械＋AEC＋VAD＋自己音声ゲート** にするのが堅いです。

```text
TTS出力音声 ─┬─> スピーカ
             └─> AEC reference

マイク入力 ─> AEC ─> VAD ─> Barge-in判定 ─> ASR ─> Dialog
```

## 1. まず自己音声を物理・信号処理で消す

### 必須

* **Acoustic Echo Cancellation / AEC**

  * TTSの再生PCMを reference としてAECに渡す
  * マイク信号からシステム音声成分を除去する
* **スピーカ音量・マイク配置の最適化**

  * スピーカとマイクを離す
  * 指向性マイクを使う
  * 可能ならイヤホン/ハンドセット運用

barge-inにはエコーキャンセラと音声区間検出が重要、という整理は一般的です。([株式会社AI Shift][1])

## 2. 状態機械で入力を扱う

最低限この4状態に分けると事故が減ります。

| 状態                   |         ASR |   barge-in |
| -------------------- | ----------: | ---------: |
| `IDLE`               |          ON |         不要 |
| `LISTENING`          |          ON |         不要 |
| `SPEAKING`           | 原則OFFまたは低感度 |         ON |
| `BARGE_IN_CANDIDATE` |        一時ON | 条件成立でTTS停止 |

ポイントは、**SPEAKING中の音声を即ASR確定に入れない** ことです。

## 3. barge-in判定は「一発停止」にしない

おすすめ条件：

```text
SPEAKING中に
  AEC後信号のVADが 300〜500ms 継続
  かつ TTS再生音との相関が低い
  かつ 音量がしきい値以上
なら barge-in candidate

ASR partial が一定信頼度以上なら TTS停止
ASR final が低信頼度なら TTS再開
```

IBM Voice Gateway でも、barge-in時にTTSを停止し、最終発話の信頼度が低い場合は再生を再開する構成が説明されています。([IBM][2])

## 4. 自己音声ゲートを入れる

TTS出力中は、マイク入力に対して以下をチェックします。

* TTS再生波形との類似度
* TTS開始からの遅延窓、例：50〜300ms
* ASR結果がTTS文と近すぎないか
* AEC residual が大きすぎないか

TTSとASR結果がほぼ同じなら、ユーザ発話ではなく **自己ループ** とみなして破棄します。

## 5. 実装方針

### 小さく始めるなら

* WebRTC AEC / RNNoise / Silero VAD などを使う
* TTS再生中はASRへ直送しない
* barge-in候補だけASRへ送る
* ASR partialでTTSを止める
* finalが無効ならTTSを再開する

### 疑似コード

```python
if state == "SPEAKING":
    mic_clean = aec.process(mic_frame, tts_reference_frame)
    speech = vad(mic_clean)

    if speech and not looks_like_tts_echo(mic_clean, tts_reference):
        barge_counter += frame_ms
    else:
        barge_counter = 0

    if barge_counter >= 400:
        state = "BARGE_IN_CANDIDATE"
        asr.start(mic_clean_buffer)

if state == "BARGE_IN_CANDIDATE":
    partial = asr.partial()

    if partial.confidence > threshold:
        tts.stop()
        state = "LISTENING"
    elif timeout_or_low_confidence:
        tts.resume()
        state = "SPEAKING"
```

## 判断基準

**音声UX重視**なら barge-in を許可。
**誤作動を避けたい**なら、TTS中はASRを止めて、wake word / push-to-talk / DTMF 的な明示操作に寄せる。

現実的には：

> AEC + VAD + confidence付きbarge-in + 低信頼度時のTTS再開

が一番バランスいいです。
